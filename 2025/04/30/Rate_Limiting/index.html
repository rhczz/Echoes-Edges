分布式限流——Redis + Lua实现滑动窗口算法

2025-04-30T10:34:26.000Z2025-04-30T10:34:26.000Z陕西西安

在调用第三方平台的 API 时，对方往往会设置调用频率的限制，若未对请求频率进行控制，可能会因超出对方设定的阈值而导致调用失败，甚至被封禁。​因此在调用端程序中需要控制对接口的访问频率。

对于单节点应用，限流的实现相对简单，常见的工具如 Guava 提供了现成的解决方案。​然而，在分布式架构中，单节点的限流无法感知其他节点的状态，容易导致全局 QPS 超限的问题。对于分布式场景下的频率限制，业内已经有很多非常成熟，非常可靠的解决方案。

但是有些时候只想通过简单的手段，实现一个轻量的频率限制。本文将介绍在redis中通过lua脚本实现滑动窗口对频率限制的实现方法。

前提条件

  • Java 17+ 、 Spring Boot 3+
  • 已部署可用的Redis实例（单机或集群）
  • 基础Lua语法

为什么选择 Lua 脚本限流

  1. 原子性保障，防止竞态条件

    Lua 脚本在 Redis 中作为一个整体执行，确保了操作的原子性。​这意味着在脚本执行期间，其他客户端的命令不会插入执行，从而避免了竞态条件。例如，在高并发的频率限制场景中，多个客户端可能同时检查和更新计数器，若使用单独的 Redis 命令，可能导致计数器更新不准确。而 Lua 脚本可以将检查和更新操作封装在一起，确保数据一致性。

  2. 减少网络往返

    在传统的客户端-服务器模型中，每个 Redis 命令都需要一次网络往返。​当需要执行多个命令时，这种方式会增加网络延迟。​使用 Lua 脚本可以将多个操作封装在一个脚本中，通过一次网络请求发送到 Redis 服务器，减少了网络往返次数，提高了执行效率。

  3. 服务器端执行，降低客户端复杂度

    将复杂的逻辑移至 Redis 服务器端执行，可以减轻客户端的负担，简化客户端代码。​例如，在实现滑动窗口限流算法时，涉及多个步骤，如添加请求时间戳、移除过期请求、检查请求数量等。​将这些操作封装在 Lua 脚本中，可以使客户端只需发送一次请求，减少了客户端与 Redis 之间的交互复杂度。

  4. 脚本缓存与重用

    Lua 脚本可以预加载到 Redis 中，并通过脚本的 SHA1 摘要进行调用，避免了每次执行时传输完整脚本的开销。​这不仅提高了执行效率，还增强了脚本的可重用性和维护性。

  5. 灵活性和扩展性

    Lua 脚本提供了编程语言的灵活性，可以实现复杂的逻辑，如条件判断、循环、错误处理等。​这使得在实现高级频率限制策略（如滑动窗口、令牌桶等）时更加灵活和高效。

滑动窗口限流算法原理

滑动窗口限流记录每次请求的时间戳，并在固定长度的时间窗口内计算请求总数：

  1. 将当前请求的时间戳和唯一 ID（可用雪花算法生成）组成有序集合（Sorted Set）的成员；
  2. 移除窗口外的过期时间戳；
  3. 统计集合大小，与阈值比较，决定是否允许；
  4. 设置 Key 的过期时间，避免冷流时大量过期成员残留。

相较于“固定窗口”算法，滑动窗口能平滑分布请求流量，避免临界点短时间内流量突增。

注意事项

  • Key 过期管理：确保 PEXPIRE 时间至少等于窗口大小，否则可能误删尚在窗口内的数据。
  • 时间同步：Redis 服务器与客户端时间需同步，否则影响时间窗口计算。
  • 性能考量：高频调用时，Lua 脚本执行可能成为瓶颈，需监控 Redis 负载。
  • 脚本长时间执行：Lua 脚本为原子操作，运行中不可中断，避免逻辑复杂度过高导致阻塞 Redis 服务。
  • 多脚本共存：可将不同限流策略拆分为独立脚本，通过 SHA1 调用，减少代码耦合。
  • 雪花 ID 或 UUID：生成 unique_id 时要确保全局唯一，防止成员冲突。
  • 二次限流：可在业务端再加一层固定窗口限流，防止因脚本失效或网络抖动造成短时间内击穿接口。
