微服务中各种认证方式

微服务中各种认证方式

基础认证（Basic Authentication）

HTTP 基本认证（Basic Authentication）是一种在 HTTP 协议层面进行用户身份验证的简易方案，通过将用户名和密码以“username:password”格式拼接并进行 Base64 编码后，放在请求头 Authorization 字段中进行传输，实现对受保护资源的访问控制。该机制最早定义于 HTTP/1.0 规范（RFC 1945），后在 RFC 7617 中进行了更新，并得到主流浏览器的广泛支持。由于 Base64 编码并非加密，Basic 认证本身无法保证凭证机密性，通常与 HTTPS 配合使用以防止中间人窃听。在小型私有系统、REST API 或临时调试场景中，Basic 认证因其实现简单、无状态且易于配置而仍被广泛采用。

Basic认证流程说明：

  1. 用户提供用户名和密码。
  2. 客户端对用户名和密码进行Base64编码，请求资源服务器。
  3. 资源服务器验证。

概述

  • 定义与标准： Basic 认证由 IETF 在 RFC 7617 中定义，属于 HTTP 认证框架的一部分，指定了凭证的拼接、编码与头字段格式。

  • 凭证格式： 客户端将用户名:密码拼接后，依据服务器指示的字符集（通常为 UTF-8）编码成字节序列，再进行 Base64 编码，最终形成类似Authorization: Basic QWxhZGRpbjpvcGVuc2VzYW1l的请求头。

  • 使用场景： 常见于保护简单资源、临时接口测试以及小规模内网应用，不依赖 Cookie、会话 ID 或登录页面，可与现代框架和 HTTP 客户端库无缝结合。

工作原理

  1. 客户端发起请求，未包含认证信息，服务端返回401 Unauthorized响应，并在WWW-Authenticate头中声明Basic realm="Protected"，指示客户端提供凭证。
  2. 客户端根据域（realm）提示，向用户收集或利用缓存的用户名和密码，并按照 RFC 7617 规定进行拼接、字符集编码及 Base64 编码，构造Authorization请求头。
  3. 客户端重试请求，将Authorization: Basic <credentials>加入请求头，服务端验证后返回目标资源或再次返回 401。

安全性

  • 明文风险： Base64 仅作编码，不具备加密或散列功能，凭证可被轻易解码为明文，若在 HTTP 明文通道中传输，易遭中间人窃听。

  • 推荐实践： 强制在 TLS/HTTPS 通道下使用 Basic 认证，确保凭证在传输过程中被加密保护；否则，认证方案等同于明文密码传输。

  • 凭证缓存与登出： 多数浏览器会缓存 Basic 认证凭证，直到浏览器或标签页关闭，HTTP 协议本身不提供登出机制，需依赖浏览器或服务器辅助策略清除缓存。

优缺点

优点 缺点
实现和配置简单，无需额外状态管理（如 Cookie 或会话） 凭证仅 Base64 编码，易被解码，需 HTTPS 保驾护航
原生支持 HTTP 协议头，主流浏览器和 HTTP 客户端库均自带支持 浏览器缓存机制不统一，无法通过协议指令强制清除或登出
适合小型私有系统、测试环境和轻量级 API 保护 不支持多因素认证、抗重放攻击能力弱，推荐使用更安全的 OAuth、JWT 或 WebAuthn
